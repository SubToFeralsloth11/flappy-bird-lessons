Lesson 2: The Game Loop

Making the bird move -- on its own, every frame.

From still image to animation

In Lesson 1 we drew a single frame. But a game isn't a painting -- it's a movie. To create the illusion of movement, we need to redraw the canvas over and over with the bird in slightly different positions each time.

The browser gives us a perfect tool for this: requestAnimationFrame. It says to the browser: "Before you paint the next screen frame, call this function." The browser calls it roughly 60 times per second.

The game loop is just a function that calls itself. Each call = one frame. In each frame we: (1) update positions, (2) clear the canvas, (3) redraw everything. Repeat. That's it.

Step 1: A function called update

1 We store the bird's position in variables and change them every frame. 
let birdY = 300;           // where the bird is vertically
let birdVelocity = 0;      // how fast it's moving (positive = down)
const gravity = 0.4;       // how much velocity increases each frame

function update() {
  birdVelocity += gravity;   // gravity pulls the bird down
  birdY += birdVelocity;     // velocity changes position

  requestAnimationFrame(update);  // schedule the next frame
}

let declares a variable that can change. (Remember const from Lesson 1? That's for things that never change.) birdY starts at 300, and each frame we add birdVelocity to it. But birdVelocity itself grows because of gravity -- so the bird falls faster and faster.

Why update only, no draw? If we call requestAnimationFrame(update) at the end of update, it creates a loop -- update calls itself 60 times a second. The draw happens separately, also 60 times a second. This is called separation of concerns.

Step 2: A function called draw

2 We clear the canvas completely, then redraw the bird at its new position. 
function draw() {
  ctx.clearRect(0, 0, canvas.width, canvas.height);  // wipe the slate

  ctx.fillStyle = '#a8d8ea';
  ctx.fillRect(0, 0, canvas.width, canvas.height);   // sky

  ctx.fillStyle = '#f1c40f';
  ctx.fillRect(180, birdY, 40, 40);                  // bird at current Y

  requestAnimationFrame(draw);  // schedule next draw
}

clearRect erases a rectangle -- in this case, the entire canvas. Without it, old frames pile up and you see a smeared mess.

We run update and draw independently. Both loop at 60fps via requestAnimationFrame. We kick them off once from requestAnimationFrame(update) and requestAnimationFrame(draw).

Try it yourself

  • Open the browser console (F12, then click Console) and type gravity = 0.05 while the demo is running -- watch the bird drift down slowly
  • What happens if gravity is 0? (The bird floats forever)
  • What happens if gravity is negative? (The bird flies upward -- anti-gravity!)
  • Change birdVelocity = -8 in the console while falling -- the bird bounces up, then falls again

Key takeaways

  • requestAnimationFrame creates a 60fps loop -- the heart of every browser game
  • Variables (let) store changing state like position and velocity
  • Gravity + velocity + position = realistic falling motion
  • Always clearRect the whole canvas before drawing each frame
  • Keep update (math) separate from draw (painting)
